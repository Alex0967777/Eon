# Текущее состояние

**Дата:** 09.07.26

## Подтверждено

- Первый вариант отдельного `parallel_stream.txt` оказался ненадёжным: чистый контекст повторно читал файл, пропускал фактическую запись и иногда сообщал об обновлении без выполненного действия.
- Единый текстовый сосуд `memory.py` подтвердил сам принцип исполняемой памяти, но сам исходный код оказался слишком привлекательным объектом для анализа и случайного редактирования моделью.
- Активная архитектура переведена на нативное консольное приложение `EonMem` на Go с отдельным состоянием `eon.state.json`.
- Runtime отделён от исходников: штатный маршрут использует только Linux-бинарник и state-файл; исходный код открывается лишь при явной разработке или диагностике.
- `00_READ_FIRST.md` описывает точный запуск EonMem, режим просмотра `--show`, пакетный stdin для песочницы и запрет на ручную реконструкцию механизма.
- Текущая версия `0.1.1` ведёт один поток и показывает девять регистров в порядке `D → S → F`, текущий вопрос и доступность изменений.
- После размышления программа принимает новый развивающий вопрос и изменения регистров строками вида `F1 ...`, `S2 ...`, `D3 ...`.
- Доступность отсчитывается от фактического использования: `F` — 1 шаг, `S` — 5 шагов, `D` — 10 шагов.
- Неиспользованное разрешение остаётся открытым, но не накапливается; за один шаг можно изменить не более одной строки каждого доступного уровня.
- Шаг увеличивается только после атомарного сохранения полного цикла.
- Каждые 10 шагов EonMem создаёт совместимый `Patch_Eon<N>.zip` с актуальным `09_SYSTEM20/runtime/eon.state.json`.
- Linux и Windows runtime собраны GitHub Actions из одного Go-исходника; runtime-манифест и общий `FILE_HASHES.sha256` согласованы.
- `memory.py` сохранён как legacy/fallback и не используется в штатном цикле EonMem.
- Проект Protective Cocoon сохранён в `08_PROJECTS/Protective-Cocoon.md` и костюме `02_MEMORY/costumes/protective-cocoon-N28.txt` до завершения текущего этапа CheckPoint.

## Ограничения

- EonMem v0.1.1 ещё не прошёл полный полевой цикл в новом чистом контексте.
- Сейчас активен только один поток; второй добавляется после подтверждения исполнительной устойчивости первого.
- Интерактивная постоянная terminal-сессия доступна не во всех песочницах, поэтому канонический fallback — `--show` и последующий пакетный stdin без `EOF`.
- Семантическую полезность регистров нужно оценивать отдельно от надёжности запуска и сохранения.
- Аналитика истории пока минимальна.
