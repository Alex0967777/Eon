# EonMem — System 20 multipart transport-safe capsule

## Назначение

EonMem отделяет жёсткую механику памяти от смыслового поля. Штатный маршрут не читает исходник, не редактирует state вручную и не создаёт альтернативную реализацию из контекста.

Текущая версия механизма: `0.2.0`.

## Структура

- `runtime/eon.loader.py` — минимальный loader capsule v3;
- `runtime/eon.runtime.json` — малый манифест `gzip+base64-parts`;
- `runtime/eon.runtime.part01.json` … `part04.json` — четыре небольшие части payload;
- `runtime/eon.state.json` — малое активное состояние format v2;
- `runtime/SHA256SUMS` — контрольные суммы неизменяемых loader и capsule;
- `history/<bucket>/step-<N>.json` — append-only запись каждого завершённого шага;
- `source/eonmem_python.py` — канонический исходник только для явной разработки;
- прежние Go/Windows runtime-файлы — legacy, не штатный маршрут.

## Почему multipart capsule v3

В v1 Base64 payload находился в одной длинной JSON-строке, а полная история накапливалась внутри `eon.state.json`. GitHub-инструмент мог обрезать оба файла ещё до запуска loader.

V2 уменьшил active state, но единый `eon.runtime.json` размером около 8 KiB всё ещё мог быть обрезан общим лимитом ответа GitHub-инструмента. V3 устраняет и этот предел:

1. `eon.runtime.json` содержит только малый манифест;
2. payload разнесён по четырём отдельным JSON-файлам менее 4 KiB каждый;
3. внутри каждой части Base64 разбит на строки по 256 символов;
4. active state не содержит полной истории;
5. каждый шаг хранится отдельным небольшим JSON-файлом;
6. workflow механически проверяет пределы каждого транспортного файла.

## Capsule v3

Loader принимает только:

- `format = EonRuntimeCapsule`;
- `version = 3`;
- `engine = python3`;
- `codec = gzip+base64-parts`.

Манифест перечисляет четыре фиксированных part-файла. Loader проверяет их формат и порядок, соединяет короткие chunks только в памяти, декодирует gzip, сверяет `source_sha256` и выполняет точный исходник. На диск исходник не записывается.

## Active state v2

`eon.state.json` содержит:

- `format_version = 2`;
- `step`;
- `registers`;
- `flow_1`;
- `last_used_step`;
- `history_index`.

Полная история не нужна для продолжения мысли и не участвует в bootstrap.

## Пошаговая история

После каждого успешного шага программа создаёт файл вида:

```text
history/000000-000099/step-000074.json
```

Файл содержит reflection, next question, UTC-время и фактические изменения регистров. Путь формируется детерминированно и не требует чтения предыдущих history-файлов.

Сохранение выполняется как восстанавливаемая транзакция:

1. запись шага готовится в `.pending`;
2. active state сохраняется атомарно;
3. pending-файл переименовывается в окончательный;
4. при следующем запуске незавершённая финализация восстанавливается автоматически.

## Один смысловой шаг

Программа показывает девять регистров в порядке `D → S → F`, доступность уровней, текущее размышление и вопрос. Затем принимает:

1. размышление до 1000 Unicode-символов;
2. вопрос до 100 символов;
3. не более одного изменения каждого доступного уровня;
4. `N`, если остальные уровни не используются.

Ритмы:

- F — 1 шаг;
- S — 5 шагов;
- D — 10 шагов.

Значение регистра ограничено 160 символами.

## Запуск

```bash
python3 ./eon.loader.py --show
python3 ./eon.loader.py < input.txt
```

После сохранения программа печатает два точных пути для GitHub-синхронизации:

- `09_SYSTEM20/runtime/eon.state.json`;
- новый `09_SYSTEM20/history/.../step-<N>.json`.

## Checkpoint

Каждые 10 шагов создаётся `Patch_Eon<N>.zip`, совместимый с `Apply-EonMemory`. В checkpoint входят active state и history-файл текущего шага. При обычной работе оба файла синхронизируются с GitHub после каждого ответа.

## Разработка

Разработка допускается только после команды пользователя `РАЗРЕШАЮ ДИАГНОСТИКУ SYSTEM 20`.

Последовательность выпуска:

1. изменить `source/eonmem_python.py`;
2. проверить на копии active state;
3. проверить запись state + history и recovery pending;
4. собрать capsule v3 с `gzip(..., mtime=0)`, четырьмя part-файлами и чанками по 256 символов;
5. проверить loader `--version` и `--show`;
6. проверить транспортные бюджеты;
7. обновить `SHA256SUMS`, workflow и документацию;
8. выпустить явный патч.
