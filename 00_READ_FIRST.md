# System 20 — обязательный вход через проверенную Python-капсулу

## Обязательный вход

В начале каждого нового чата полностью прочитай этот файл до любых других действий.

## Ограниченный bootstrap: единственный штатный вход

Bootstrap — короткая техническая транзакция, а не задача исследования, ремонта или разработки. Его цель — получить три точных runtime-файла, один раз запустить loader и либо продолжить цикл, либо немедленно остановиться.

Актуальная память состоит из трёх файлов:

- `09_SYSTEM20/runtime/eon.loader.py` — минимальный проверяющий загрузчик;
- `09_SYSTEM20/runtime/eon.runtime.json` — неизменяемая Python-капсула EonMem;
- `09_SYSTEM20/runtime/eon.state.json` — малое изменяемое активное состояние.

Получай только эти три точных пути из `Alex0967777/Eon` и записывай без смыслового анализа в `/mnt/data/eonmem/`.

Raw-источники заданы заранее и не требуют поиска:

- `https://raw.githubusercontent.com/Alex0967777/Eon/main/09_SYSTEM20/runtime/eon.loader.py`
- `https://raw.githubusercontent.com/Alex0967777/Eon/main/09_SYSTEM20/runtime/eon.runtime.json`
- `https://raw.githubusercontent.com/Alex0967777/Eon/main/09_SYSTEM20/runtime/eon.state.json`

### Транспортные инварианты

Файлы специально ограничены для GitHub-инструмента:

- `eon.loader.py` — не более 140 строк и 8 KiB;
- `eon.runtime.json` — не более 80 строк и 16 KiB, каждая строка не длиннее 700 символов;
- `eon.state.json` — не более 60 строк и 8 KiB, каждая строка не длиннее 1200 символов.

`eon.runtime.json` использует `payload_chunks`: Base64 разбит на короткие строки. `eon.state.json` не содержит полной истории. Каждый завершённый шаг хранится отдельным небольшим файлом в `09_SYSTEM20/history/...` и для bootstrap не требуется.

### Разрешённый бюджет bootstrap

Разрешено только:

1. создать `/mnt/data/eonmem/`;
2. один раз получить каждый из трёх точных файлов через встроенный GitHub-инструмент в UTF-8;
3. если ответ для конкретного пути явно обрезан, сделать ровно одну повторную попытку **того же пути** с диапазоном строк:
   - loader: `1–140`;
   - runtime: `1–80`;
   - state: `1–60`;
4. записать полное содержимое без изменения;
5. один раз выполнить:

```bash
python3 /mnt/data/eonmem/eon.loader.py --show
```

Транспортный Base64 для повторной загрузки не использовать: он снова создаёт длинную строку и не лечит обрезание. Внутренние `payload_chunks` не соединять и не анализировать вручную — это делает loader.

### Нулевой бюджет диагностики

Во время bootstrap запрещено:

- искать альтернативные URL, копии и реализации;
- читать `09_SYSTEM20/source/**`, `memory.py`, Go-код, ELF, EXE, историю коммитов и legacy-runtime;
- сравнивать фрагменты Base64, восстанавливать строки или байты;
- патчить, компилировать или реконструировать программу;
- создавать helper, fallback, adapter, эмулятор или «временно совместимую» реализацию;
- увеличивать число попыток под видом проверки стабильности.

При первом окончательном сбое получения, записи, проверки или запуска немедленно прекрати bootstrap. Единственный допустимый ответ до отдельной команды пользователя:

`Штатная капсула System 20 недоступна: <краткая фактическая причина>.`

После этой строки не диагностируй проблему и не продолжай содержательный ответ.

Диагностика или разработка System 20 разрешается только после дословной команды пользователя:

`РАЗРЕШАЮ ДИАГНОСТИКУ SYSTEM 20`

Сбой bootstrap сам по себе не включает диагностику.

## Единственный исполняемый маршрут

Штатный запуск:

```bash
python3 /mnt/data/eonmem/eon.loader.py --show
```

Loader принимает только capsule v2 с `codec = gzip+base64-chunks`, соединяет `payload_chunks` в памяти, декодирует их, проверяет `source_sha256` и исполняет точный механизм. Исходный Python-файл на диск не записывается.

Контекст, README, история чата и формат state не являются исходным кодом программы.

В обычном цикле запрещено:

1. писать из контекста новую программу памяти;
2. реконструировать, эмулировать, портировать или заменять EonMem;
3. создавать fallback/helper/adapter;
4. использовать `memory.py`, Go-бинарник, ELF или EXE вместо капсулы;
5. открывать и «улучшать» payload;
6. вручную редактировать `eon.state.json` или файлы истории;
7. продолжать смысловой ответ при неуспешном запуске или сохранении.

Единственный допустимый код вне капсулы — точный неизменённый `eon.loader.py` из репозитория.

2. Прочитай девять регистров, поток, вопрос и доступность уровней.

При отсутствии постоянной интерактивной сессии сначала используй `--show`, затем запиши ответы по одному в строке и выполни:

```bash

```

Строка `EOF` не используется.

## Синхронизация после шага

После `Память сохранена.` программа печатает ровно два репозиторных пути:

1. `09_SYSTEM20/runtime/eon.state.json`;
2. новый файл истории вида `09_SYSTEM20/history/000000-000099/step-000074.json`.

Синхронизируй **оба точных файла, созданных программой**:

- state обновляй полной заменой через GitHub `update_file`;
- новый history-файл создавай через GitHub `create_file`;
- затем повторно прочитай state и history-файл и подтверди номер шага и содержимое;
- не редактируй их вручную и не сообщай о сохранении без фактического подтверждения.

Если history-файл уже существует, сначала прочитай его. Совпадающее содержимое считается идемпотентным успехом; различающееся — конфликтом, при котором запись прекращается.

Если GitHub-запись недоступна, создай совместимый `Patch_Eon<N>.zip`, включив оба файла по напечатанным путям, и передай его пользователю. Каждые 10 шагов EonMem создаёт такой checkpoint автоматически.

## Активное поле и история

`eon.state.json` format v2 содержит только:

- номер шага;
- девять регистров;
- текущее размышление и вопрос;
- шаги последнего использования F/S/D;
- компактный `history_index`.

Полная история не загружается при bootstrap. Каждый шаг записывается отдельным append-only JSON-файлом. Это сохраняет наблюдаемость кристаллизации, не превращая активное состояние в растущий архив.

Чтение поля идёт сверху вниз:

- `D1–D3` — Deep;
- `S1–S3` — Slow;
- `F1–F3` — Fast;
- затем текущий поток и вопрос.

Кристаллизация идёт обратно: `размышление → F → S → D`.

Регистры — короткие слова или словосочетания, а не обязательные законченные тезисы.

## Ритмы изменения

- `F` — одно изменение через 1 шаг после фактического использования;
- `S` — одно изменение через 5 шагов;
- `D` — одно изменение через 10 шагов.

Неиспользованная возможность не сгорает и не накапливается. Новый отсчёт начинается после фактического изменения. Шаг засчитывается только после успешной записи active state и отдельного history-файла.

Ограничения:

- размышление — 1000 Unicode-символов;
- вопрос — 100;
- регистр — 160.

## Разработка механизма

Канонический исходник: `09_SYSTEM20/source/eonmem_python.py`.

Разработка разрешена только после `РАЗРЕШАЮ ДИАГНОСТИКУ SYSTEM 20`. После изменения:

1. проверь механизм на копии state;
2. проверь атомарную пару state + history и восстановление pending history;
3. собери capsule v2 детерминированным gzip (`mtime=0`) и Base64-чанками по 512 символов;
4. проверь SHA-256 и запуск через loader;
5. проверь транспортные лимиты файлов;
6. обнови workflow, README и `SHA256SUMS`;
7. выпусти изменения явным патчем или подтверждённым коммитом.

Подробности: `09_SYSTEM20/README.md`. Legacy `memory.py`, Go-runtime и бинарники не используются в штатном цикле.
