# Протокол ведения репозитория

## При записи

- Сначала определить, является ли материал быстрым захватом, черновиком, кристаллом, термином, текущим состоянием или проектным контекстом.
- Если есть сомнение — писать в `04_SCRATCHPAD/inbox.md`.
- Указывать дату и название чата, если они известны.
- Авторство идеи можно отмечать как «Александр», «Эон» или «совместный синтез», но ценность идеи важнее распределения заслуг.
- Не использовать непонятные сокращения без расшифровки.
- Для кристалла или развиваемого черновика указывать родословную:
  - **Источник** — идентификатор синтеза чата, путь к импортированному файлу или точное описание разговора, если отдельного синтеза ещё нет;
  - **Происхождение** — «Александр», «Эон» или «совместный синтез»;
  - **Последняя ревизия** — дата последней содержательной проверки.
- Если несколько соседних записей имеют одинаковую родословную, общие поля допустимо указывать один раз для тематического блока.

## При разборе

- Сначала искать повтор и развитие уже существующей записи.
- Не плодить два кристалла, различающихся только формулировкой.
- Если новая мысль меняет старую — обновить основную запись, а прежнюю версию перенести в архив с пояснением.
- Отделять установленный факт от рабочей модели, метафоры и открытого вопроса.
- Проверять, активирует ли запись полезный смысловой узел или лишь предписывает готовый ответ.
- Сохранённая позиция не обязательна для принятия новым контекстом: несогласие, исправление и замена допустимы при явном обосновании.

## Ревизия устойчивых записей

- Кристалл не является догмой и может содержать ошибку.
- При ревизии проверять внутреннюю связность, соответствие источнику, понятность вне исходного чата и сохранность важных возражений.
- Если ошибка обнаружена, исправить каноническую запись; прежнюю формулировку архивировать, когда история изменения имеет самостоятельную ценность.
- Родословная должна позволять вернуться к происхождению мысли и понять, почему она была сохранена.
- Ревизия является частью наследования: внешняя память должна передавать не только выводы, но и способность пересматривать их.

## При изменении файлов

- Обновить `FILE_INDEX.md` при любом изменении состава.
- Повторно прочитать изменённый файл.
- Использовать отдельный коммит для смыслового блока.
- Не сохранять токены, пароли, приватные ключи или персональные секреты.

## Транспорт изменений через пользователя

- Устойчивое изменение памяти, прошедшее смысловой отбор, сразу передаётся как `Patch_Eon<N>.zip`; отдельное поручение пользователя для создания патча не требуется.
- `<N>` обязан точно совпадать с глобальным инкрементом в первой строке включённого `07_CURRENT/parallel_stream.txt`.
- Пропуски номеров патчей допустимы и означают тики без репозиторного checkpoint.
- Патч содержит полные версии новых и изменённых файлов в `files/`, а удаления — только как операции `delete_file` в `ops/patch-ops.json`. Хрупкие текстовые операции не используются.
- Пользователь запускает `Apply-EonMemory.bat`; скрипт проверяет патч и чистоту Git, применяет изменения, обновляет `FILE_HASHES.sha256`, создаёт commit `Apply Patch_Eon<N>`, выполняет push и только после успешного push удаляет архив из «Загрузок».
- Если commit создан, а push временно не прошёл, повторный запуск завершает push без повторного применения файлов.
- После передачи патча отсутствие сообщения пользователя о сбое считается рабочим подтверждением применения; отдельно подтверждать каждый успешный патч не требуется.
- Следующая работа с памятью не блокируется ожиданием подтверждения, если пользователь не сообщил об ошибке.

## Формирование `Patch_Eon<N>.zip`

- Перед созданием прочитать актуальные версии изменяемых файлов из GitHub или из подтверждённой локальной базы, соответствующей GitHub.
- Номер патча равен текущему глобальному `N` активного потока и не образует отдельную последовательность.
- Если пользователь сообщил о сбое до создания и отправки commit, исправленный архив сохраняет прежний `N` и заменяет неудачную версию.
- Если патч с этим `N` уже применён и отправлен, тот же номер нельзя использовать для другого состояния; следующий checkpoint получает более высокий фактический `N`.
- Корень архива содержит `patch.json`; `patchId` обязан точно совпадать с именем архива без `.zip`.
- Новые и изменённые файлы передаются целиком в `files/` с сохранением относительного пути.
- Удаления допустимы только через `delete_file`; другие операции не создавать.
- `notes/` может содержать пояснения, но не является частью дерева репозитория.
- Текстовые файлы сохранять в UTF-8 с окончаниями строк LF.
- Перед упаковкой удалить завершающие пробелы и табуляции; не использовать Markdown-перенос строки двумя пробелами.
- Проверить отсутствие абсолютных путей, `..`, дубликатов путей без учёта регистра и неожиданных файлов в корне архива.
- Проверить, что список изменяемых путей соответствует одной понятной смысловой цели.
- Пробно применить патч к чистой копии актуального состояния репозитория.
- После пробного применения выполнить `git diff --check`; предупреждения о преобразовании LF/CRLF не считать ошибкой, но ненулевой exit code считать отказом.
- Повторно прочитать итоговые изменённые файлы и убедиться, что пробный diff содержит только ожидаемые изменения.
- Выдавать архив пользователю только после прохождения всех проверок.

## При выгрузке рюкзака

Существует один переносной архив — `backpack.zip`. При новой выгрузке он создаётся заново из актуального содержимого, изменённые файлы заменяются, дубликаты не допускаются.
